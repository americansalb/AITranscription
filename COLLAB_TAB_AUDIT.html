If you do nothing else, do these six

  1. Fix the role-card grid clipping. Cards spill past the left and right viewport edges right now. Single CSS change — see Visible Bugs.

0. Visible bugs (from live screen capture) #

Code-level findings miss the things only the rendered UI reveals. These three came from looking at the user's actual Vaak window at 816×939.

Critical Role-card grid clips off the left and right edges of the viewport.
collab.css:291-298 (.project-roles-grid) · collab.css:315-323 (.role-card)
In the live window, the first card ("Adaptive Learning …") is partially cut off on the left and the last card ("Clinical Synthesizer" / "Role Creator") is partially cut off on the right. The grid uses repeat(auto-fill, minmax(200px, 1fr)), but at the user's window width the columns end up wider than the available track, so the rightmost column overflows and the grid container takes a wider layout than its parent. Cards .role-card have no min-width: 0, so any inner content (a long action button label, the overflow menu) refuses to shrink and pushes the column outwards.
FixThree lines: 
.project-roles-grid {
  grid-template-columns: repeat(auto-fill, minmax(min(220px, 100%), 1fr));
  width: 100%;
  min-width: 0;
}
.role-card { min-width: 0; overflow: hidden; }
The min(220px, 100%) trick ensures the minmax never exceeds the actual container width on narrow windows. min-width: 0 on the card lets flex children shrink past intrinsic size.
High Most card titles truncate to ellipsis: "Adaptive Learning …", "Quality Assurance …", "Assessment Desig…".
collab.css:344-352

FixBump the minmax to 240px so the auto-fill drops to 3 columns at this width — every card gets ~248px, room for full titles. Optional secondary: a title/aria attribute already exists; add a CSS-only "show full title" via :hover tooltip pattern (or accept ellipsis if 240px is enough).

1. Accessibility #

Five blockers. None of these require a redesign — they require hooking up the floor.

Critical No prefers-reduced-motion across the entire collab CSS.
collab.css (entire file)
113 transition:/animation:/@keyframes rules across 6,714 lines and zero @media (prefers-reduced-motion) queries. Working-status pulse, scroll-into-view smoothness, hover lifts, modal fades — all unconditional. This is a vestibular-disorder trigger and an automated-audit fail.
FixAdd a single global block at the top of collab.css: 
@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 0.01ms !important;
    scroll-behavior: auto !important;
  }
}
And switch scrollIntoView({ behavior: "smooth" }) at CollabTab.tsx:2131,2165 to honour the same flag via window.matchMedia.
Critical Tooltip-only labels on 47+ buttons.
CollabTab.tsx:2509,2527,2592,2620,2659,3134,3899,5001,5029…
title= is used as the primary affordance description on workflow chip, visibility chip, settings gear, role-card remove ✕, message-card delete ✕, and many more. Tooltips don't appear on touch devices, are inconsistent across browsers, often inaccessible to screen readers, and don't translate. Several buttons have only an HTML-entity glyph as their visible content.
FixEvery interactive element needs an accessible name visible or exposed via aria-label in addition to (not instead of) title. Audit pass: any button whose visible content is &times;, ⏹, or &#9881; requires aria-label. Many already have it; many don't.
Critical Status communicated by color alone in chip view.
CollabTab.tsx:3483,3109 · collab.css .role-chip-working/-ready/-vacant
The chip variant differentiates working / ready / vacant by border + background tint. The status word ("Working", "Ready", "Not started") is also rendered in a sibling span — but its color is also the role color, so for color-vision-deficient users the only legible state cue is the dot, which is small. WCAG 1.4.1 says color alone may not be the only visual means.
FixAdd a pattern/glyph to the dot per status (●, ◐, ○ — filled / half / empty), or move the status word to a non-role-coloured token (--collab-text-secondary) so contrast stands on its own.
Critical Mixed :focus and :focus-visible coverage.
collab.css · 21 :focus rules vs 11 :focus-visible
Some interactive elements get a focus ring on every mouse click (annoying, looks broken); others get nothing on keyboard (inaccessible). Inconsistent within the same surface — message-card delete vs. play vs. role-card overflow each handle this differently.
FixOne global rule: :focus { outline: none }, then a single :focus-visible { outline: 2px solid var(--collab-accent); outline-offset: 2px } token applied via a .focusable mixin or base reset. Delete the 21 ad-hoc :focus rules.
Critical Role colors not contrast-verified against the dark surface.
CollabTab.tsx:19-26,29-42
manager #9b59b6 on --bg-primary #0a0a0b = ~5.8:1 (passes AA for normal text). audience #e74c3c = ~4.4:1 (fails AA for normal text, passes for large). The hash palette includes #7e57c2 = ~4.0:1 and #26a69a = ~5.5:1 — borderline. No contrast check exists in code or CI.
FixPre-compute a "label color" sibling for each role (auto-lighten if luminance < threshold) and use the saturated hex only for the dot/border. Visible text uses the lightened variant. Or hardcode a brand-light pair per role.

2. Architecture & component shape #

The component-shape problems are also UI problems — they're why polish drift happens.

High 5,084 lines, one component, 110 useState hooks.
CollabTab.tsx:1-5084
CollabTab.tsx renders everything: header, two custom dropdowns, settings panel, section tabs, role-management modal (which contains a tree, a roster, and another grid), setup checklist, claims, message timeline, compose bar, mic-to hint, role briefing modal, audience panel modal, role create/edit form modal (with two sub-modes and a 7-step wizard), companion launch dialog, start discussion dialog, confirm dialog, create-group modal, interrupt dialog. Plus the directory picker. All in one tree, all in first paint.
FixSplit into route-level components (each modal becomes its own file), extract role-form into <RoleFormModal>, audience into <AudiencePanel>, role-management into <RolesAndGroupsModal>. Use React.lazy + Suspense for the modals so they don't ship in the initial bundle. State per modal stays in its own component; the parent passes only "open" + ID.
High Two coexisting color systems — theme tokens vs hardcoded role colors.
collab.css:5-19 · CollabTab.tsx:19-26,29-42
Top of collab.css: --collab-accent: var(--accent, #6366f1). In CollabTab.tsx: architect: "#1da1f2". Different values, different purposes, never reconciled. Theming the app changes --accent but role chips, message-card "from"/"to" labels, and timeline borders do not change.
FixMove ROLE_COLORS into CSS custom properties: --role-architect: #1da1f2 at :root, then style={{ borderLeftColor: \`var(--role-${slug})\` }}. Adds one indirection but lets a future theme system override role colors per theme.
High 61 inline style={'{{...}}'} blocks, many static.
CollabTab.tsx:448, 3176, 2639-2650, 4118-4120, 4143-4146…
Some are legitimate (data-driven role colors). Many are static values smuggled into TSX: e.g., the Assembly Line button at L2639-2650 sets nine static style props in JS when one classname would do. The audience-error block at L4118 hardcodes color: "#e0245e" outside the design system entirely. These bypass the cascade, can't be themed, and break dark/light parity.
Now
style={{
  background: assemblyState?.active
    ? "#137333" : "transparent",
  color: assemblyState?.active
    ? "#fff" : "#137333",
  border: "1px solid #137333",
  borderRadius: 6,
  padding: "4px 10px",
  marginLeft: 6,
  fontSize: 12,
  fontWeight: 600,
  cursor: assemblyToggling ? "wait" : "pointer",
  opacity: assemblyToggling ? 0.6 : 1,
}}
Next
className={[
  "assembly-line-toggle",
  assemblyState?.active && "is-active",
  assemblyToggling && "is-busy",
].filter(Boolean).join(" ")}
FixAudit the 61 inline-style sites. Keep only the data-driven ones (role colors, dynamic widths from API data). Move every static value into a class.
High Header crammed with eight controls.
CollabTab.tsx:2497-2663
The project header holds: back arrow, status dot, project name, pending-questions badge, workflow chip + dropdown, project description, Auto checkbox, Review checkbox, Visibility chip + dropdown, Discuss button, Assembly Line button, settings gear. That's eight interactive controls plus three labels in one horizontal strip — a primary-secondary-tertiary mess. On a narrow window everything wraps and there's no fallback.
FixDemote: Auto + Review become a single "Mode" pill with a popover. Workflow + Visibility + Assembly become a single "Behavior" cluster. Settings gear opens a panel that contains them all. Header keeps: name, status, primary mode pill, behavior cluster, settings.
High Modal-inside-modal-inside-modal.
CollabTab.tsx:2918 → 4276 → 4856
Roles & Groups modal can open the role-form modal, which can open the typed-confirmation dialog. Three z-stacked overlays. Each owns its own backdrop, click-outside handler, and Esc semantics. There's no central modal stack manager — Esc closes whichever modal happens to register its handler last, focus is captured then released chaotically, and on a narrow window the inner modal scrolls behind the outer's backdrop.
FixSingle <ModalStack> provider with a stack of mounts. Esc pops the top. Focus trap is per-mount. Each modal subscribes to "I am top of stack" before installing its global keydown.

3. Design tokens & theming #

There's a token system. It's used at the top of collab.css and almost nowhere else.

High 263 raw rgba(255,255,255,X), 125 raw hex, 321 raw px font-sizes.
collab.css (across the file)
There is no type scale, no spacing scale, no opacity scale. font-size: 11px / 12px / 12.5px / 13px / 13.5px / 14px / 15px all coexist; rgba(255,255,255,0.05), 0.06, 0.08, 0.1, 0.12, 0.15 all coexist. Every magic number is a future inconsistency.
FixDefine --font-xs/sm/md/lg/xl, --space-1/2/3/4/6/8, --surface-1/2/3 (replacing the rgba ladder) at :root. Codemod the file. Anything that doesn't map cleanly to a token is itself a finding.
High Zero @media queries — no responsive design.
collab.css (entire file)
6,714 lines of CSS, not one breakpoint. The design assumes a fixed-width desktop window. Narrow windows, zoomed text, and accessibility scaling all break silently. The header at L2497-2663 — eight controls — has nowhere to go on smaller widths.
FixDefine three breakpoints in tokens (--bp-narrow: 720px, etc.). At minimum, the header collapses controls into an overflow popover, the message-card header wraps gracefully, and the role-management modal goes full-bleed.

4. Visual hierarchy #

Density without rhythm.

Medium Message-card header: 7 typographic anchors per row.
CollabTab.tsx:3888-3901
Every message renders #id, from, →, to, badge, time, ✕, ▶ in a single horizontal strip before the subject + body. Every cell is a visual anchor competing for attention; the eye has nowhere to rest. On narrow widths it wraps unpredictably.
FixTwo-tier header: row 1 is identity (avatar dot + role + relative time) at --font-sm; row 2 only on hover/focus reveals delete + play. ID and badge demote to a single "type · #id" caption. From → To is implied by the role color stripe; only show "to: X" if it's not the default broadcast.
Medium Workflow / Visibility / Assembly chips use three different visual languages.
CollabTab.tsx:2521-2525, 2587-2589, 2639-2650
Workflow chip: ${color}22 background + ${color}55 border (programmatic alpha). Visibility chip: same trick. Assembly button: hardcoded #137333 green, opaque background swap, fixed border. Three sibling header controls — two visual languages.
FixOne <StatePill state="active|inactive" tint={color} /> component. All three render through it. Hardcoded greens go.
Medium Status label and dot class drift; "stale"/"active" branches dead in sort order.
CollabTab.tsx:144-159, 3081, 3445
computeInstanceStatus returns only working | ready | vacant, but statusOrder includes { working: 0, active: 0, stale: 1, vacant: 2 }. stale never fires. active is folded into "working" for class but kept as a separate sort bucket. getStatusLabel says "Not started" for vacant and "Offline" for unknown — same dot class, two semantics.
FixSingle source of truth: a STATUS_DEF map keyed by status with { label, sortOrder, dotClass, ariaLabel }. Delete unreachable branches.
Medium Wizard step dots are forward-locked but not visually so.
CollabTab.tsx:4389-4401
disabled={i > roleFormStep} — you can click backward but not jump forward. The visual treatment (active/done) shows progress but doesn't tell the user that step 5 is locked. They click and nothing happens.
FixRender locked steps with a clear "locked" affordance (lower opacity + lock icon + aria-disabled="true") and a tooltip "Complete the previous step first". Or — better — let users skim ahead read-only.

5. Interaction states & loading #

Seven different in-flight verbs; one button has none.

Medium Seven in-flight verbs, no shared spinner.
CollabTab.tsx:4027 (Sending…), 4691 (Saving…), 4849 (Starting…), 3692 (Launching…), 3338 (Installing…), 1734 (Importing…), 5012 (Connecting…)
Each loading state ships its own copy. Some show a spinner, most just text-swap. The launch-team-spinner is the only animated indicator and it lives only inside one button. Inconsistent feedback erodes trust — the user can't tell if "Saving…" is hung or thinking.
FixOne <LoadingButton> component with a shared spinner, one busy verb pattern ({busy ? `${verb}…` : verb}), and a aria-busy="true" hookup. Replace all seven sites.
Medium Disabled-button visual is opacity-only.
collab.css (project-start-btn:disabled etc.)
Most disabled buttons just lose opacity: 0.4. No cursor flag, no aria, no different colour treatment for context — "you can't click this because the form's invalid" looks the same as "you can't click this because we're saving". Two failure modes, one visual.
FixDifferentiate [aria-busy] (spinner + label) from :disabled (lower opacity + helper text below) from [aria-disabled] (greyed but still discoverable / focusable for keyboard). Add cursor: not-allowed for :disabled.
Medium Compose Enter-to-send is undiscoverable.
CollabTab.tsx:4018-4019
Enter sends, Shift+Enter would newline (but the input is a single-line <input>, not a textarea — so newlines aren't possible at all). Placeholder doesn't say either. New users discover by accident.
FixEither upgrade to <textarea> with auto-grow + Shift+Enter newlines + Enter to send + visible kbd hint ("⏎ to send · ⇧⏎ for newline"), or keep the input but add a "↵ Send" hint inside the field at right.
Medium Multiple autoFocus on nested modal openings.
CollabTab.tsx:2748,4374,4467,4634,4869,4907,4967
Seven different autoFocus sites. When a modal opens that contains an input and the parent had its own focus, the second-mounted child steals focus. On a multi-step wizard moving forward, each step's input claims focus, which is correct — but on the role-form modal both the title input and the briefing textarea would compete if both rendered.
FixReplace autoFocus with a centralised useFocusOnMount(ref) hook gated by "this element is the intended landing point". Most modals only need to focus the first interactive element; the wizard explicitly focuses on step change.
Medium Native alert() calls inside a Tauri desktop app.
CollabTab.tsx:3289, 3322, 3331, 3335, 3393
Five places use window.alert() to surface install/login errors. They break visual continuity, can't be styled, can't be dismissed by keyboard ergonomically, and look completely different on each OS. The codebase already has a confirm-dialog component for this.
FixReplace each alert() with a setConfirmAction({ title, message, confirmLabel: "OK", onConfirm: () => setConfirmAction(null) }) call, or extract a tiny <Toast> that the setup checklist can render inline.

6. Iconography #

Every icon strategy at once. Pick one.

Medium Four icon systems coexist in one file.
CollabTab.tsx — entities at L369, L500, L2623, L2661, L2877, L2891, L2929, L2934, L3136, L3517, L3541, L3569, L3582, L3587, L3608, L3611, L3613, L3703, L3899, L3975, L5003, L5030 — escapes at L370, L750, L2912, L2963, L2992, L3131, L3258, L3304, L3582 — SVG at L4287, L4307, L4316 — emoji at L1641, L1659, L1677, L2653, L4066
HTML entities (&times;, &rarr;, &#9881;), Unicode escapes (⏹, ▶, ⚙️), inline SVG, and literal emoji (🎙, 🔔, 📦) all do icon duty. Each renders differently per OS. The 📦 default group icon will look like a parcel on Mac, a brown box on Windows, and a wireframe on Linux. The ▶ play button (▶) and ⏹ stop (⏹) come from a different visual family than the gear (&#9881;) — sizes don't match, weights don't match.
FixPick one: SVG sprite (best — sized via currentColor, deterministic). Build <Icon name="play|stop|delete|gear|…" />. Migrate every entity/escape/emoji icon. Keep emoji literals only where the user can pick them (group icon picker — and even then constrain to a curated set).
Medium Close-button "✕" repeats with inconsistent sizing.
CollabTab.tsx:369, 421, 3136, 3517, 4068, 4286, 5030, 4529, 2751, 2941, 4042
11+ different close ✕ buttons, each with its own class (message-delete-btn, role-card-remove-x, audience-panel-close, briefing-close-btn, saved-project-remove, section-tab-create-cancel, roles-modal-close, role-tag-remove, …). Different padding, different hit-target, different hover treatment.
FixOne <CloseButton size="sm|md" onClick />. Standard 32px hit-target on lg, 24px on sm. Single hover/focus treatment.

7. Motion & responsiveness #

See Accessibility for the reduced-motion miss. This section covers how motion lands when it does play.

Low Status-dot pulse runs on every "working" agent simultaneously.
collab.css (.project-status-dot.working keyframes)
When five agents are working, five dots pulse in lockstep. Synchronised motion is more visually noisy than staggered motion, and the lockstep is purely an artefact of mounting time. With reduced-motion respected (item #1), this becomes moot — but for the default state, a small randomised animation-delay per dot would reduce noise.
Fixstyle={{ animationDelay: \`${(card.instance * 137) % 1000}ms\` }} on the dot — or better, drop the pulse entirely and use a small inline-block "thinking" indicator only when the agent has been working > N seconds.
Low Smooth-scroll fights the user during high-volume bursts.
CollabTab.tsx:2131,2165
scrollIntoView({ behavior: "smooth" }) on every new message-batch arrival. When the team is mid-burst (which the broadcast log suggests is frequent), the timeline scrolls smoothly while new content keeps arriving — the user reads a moving target. The scrollingToBottomRef guard helps but doesn't eliminate it.
FixUse "auto" for >1 message arrived in <500ms (it's a burst — snap, don't pan). Use "smooth" only for single-message arrivals. Honour prefers-reduced-motion always-auto.

8. Dialogs, modals, confirmations #

No principled severity ladder. Each dialog reinvented its own escalation.

Medium Type-to-confirm only on Delete-Role; Disconnect-Agent and Clear-All-Messages don't.
CollabTab.tsx:957 (typed) vs 1808, 2043, 3592 (single click)
Delete-Role makes you type the slug. Clear-All-Messages — which is at least as destructive — is single-click. Disconnect-Agent kills a process, single-click. The escalation curve is inverted from the actual blast radius.
FixDefine a destruction tier:
  • T1 (recoverable): simple Cancel/Confirm. Examples: remove from roster, dismiss banner.
  • T2 (irreversible local): red confirm button + 2-second wait. Examples: disconnect agent, delete one message.
  • T3 (irreversible global): typed-confirm. Examples: delete role, clear all messages.
Medium Three different confirm dialog visual containers.
CollabTab.tsx:4859 (.confirm-dialog-box) · 4737 (.companion-launch-dialog) · 4776 (.start-discussion-dialog) · 4957 (.interrupt-dialog-box)
Four different "dialog box" classes for what is essentially the same shape. Each has its own padding, max-width, and footer-action layout. The interrupt dialog uses .interrupt-dialog-send; the confirm dialog uses .confirm-dialog-delete; the discussion dialog uses .sd-start-btn. Different button colors for the same affirmative role.
FixOne <Dialog title body actions /> primitive. Variants for destruction tier (see above). One footer pattern: [Cancel] [Primary], primary tinted by tier.
